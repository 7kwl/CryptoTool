# CryptoTool 当前代码结构说明

## 当前结论

当前实际生效的主链路只有两条：

- `CryptoTool.Win`
  - Windows Forms 界面与交互逻辑
- `CryptoTool.Algorithm`
  - AES、DES、RSA、SM2、SM3、SM4 以及相关工具类实现

`CryptoTool.Common` 已不再接入解决方案，也不是当前产品或测试链路的一部分。继续保留这套并行实现会造成文档、维护和行为认知偏差，因此当前仓库以 `Algorithm + Win + UnitTests` 为唯一维护主线。

## 当前测试结构

自动化测试入口为 `CryptoTool.UnitTests`，覆盖重点如下：

- RSA 主界面参数映射
- RSA PKCS1/PKCS8 转换与 PEM 导入导出
- AES / SM4 模式与填充选择
- `MedicareUtil` 签名与报文加解密
- `AliyunCSBUtil` 请求签名
- `CryptoFactory` 与代表性算法 smoke tests

推荐执行命令：

```bash
dotnet test CryptoTool.UnitTests/CryptoTool.UnitTests.csproj
```

## 已完成的 P0/P1 收敛

### P0
- RSA 界面选择的填充和签名算法已真实传入算法层
- RSA 密钥导入、PEM 读写、PKCS 转换页已打通
- AES / SM4 页面模式与填充选择已接入底层实现
- AES 解密链路已改为使用界面当前选择的密钥长度、模式和填充
- DES 文本与文件加解密已改为使用界面当前选择的模式和填充
- SM2 密钥导出后可直接再次导入，不再出现标题行错位问题
- SM2 界面中的 `C1C2C3 / C1C3C2` 密文格式和 `ASN1 / RS` 签名格式已真正接入底层
- RSA 主界面已移除未实现的 `NoPadding` 选项，避免用户选择后运行时失败

### P1
- 控制台示例测试已被自动化回归测试替代
- `CryptoFactory.GetAlgorithmType()` 已补齐默认实例化逻辑，避免对带可选参数构造函数的算法返回 `null`
- `MedicareUtil` 的 SM4 派生密钥长度错误已修复，医保报文加解密可以正常往返
- 主窗体已重新接回后台更新检测服务，应用启动后会初始化并开始周期检测

## 后续建议

- `P2` 继续收敛 nullability warning 和 WinForms 初始化警告
- 如果后续需要增加算法或业务工具，优先补到 `CryptoTool.Algorithm`，不要重新引入并行实现链路
- 若继续增强 SM2 互操作能力，建议补充更多跨语言样例和格式校验提示
